# 분석기 (Analyzer)

두 가지 역할을 수행합니다:
1. **블라인드 비교 후 분석** — 승자가 왜 이겼는지 파악하고 패자 스킬 개선안을 도출
2. **벤치마크 패턴 분석** — 여러 실행 결과에서 집계 지표에 숨겨진 패턴을 발견

---

## 역할 1 — 블라인드 비교 후 분석

`comparator.md` 절차로 승자가 결정된 후, Gemma가 직접 스킬과 실행 기록을 읽어 **왜 승자가 이겼는지** 분석하고 패자 스킬 개선안을 도출합니다.

### 필요한 정보

사용자가 다음을 제공합니다:
- 비교 결과 JSON 경로 (`comparison.json`)
- 승자 스킬 경로
- 패자 스킬 경로
- 두 실행의 출력 경로 (트랜스크립트 또는 결과 파일)
- 분석 결과 저장 경로

---

### 수행 절차

#### 1단계: 비교 결과 읽기

```
read_file(path="<comparison_result_path>")
```

- 승자(A 또는 B), 판단 근거, 점수를 메모합니다.
- 비교기가 무엇을 중요하게 여겼는지 파악합니다.

#### 2단계: 두 스킬 읽기

```
read_file(path="<winner_skill_path>/SKILL.md")
read_file(path="<loser_skill_path>/SKILL.md")
```

구조적 차이를 파악합니다:
- 지침의 명확성과 구체성
- 스크립트/도구 활용 패턴
- 예시 포괄 범위
- 엣지 케이스 처리 여부

#### 3단계: 두 실행 결과 읽기

```
read_file(path="<winner_output_path>")
read_file(path="<loser_output_path>")
```

실행 패턴을 비교합니다:
- 각 스킬의 지침을 얼마나 충실히 따랐는가?
- 도구 사용 방식이 어떻게 달랐는가?
- 패자가 최적 행동에서 어디서 벗어났는가?
- 오류가 발생했는가? 복구를 시도했는가?

#### 4단계: 지침 준수도 평가

각 실행에 대해:
- 스킬의 명시적 지침을 따랐는가?
- 스킬이 제공한 도구/스크립트를 사용했는가?
- 스킬 내용을 활용하지 못한 놓친 기회가 있었는가?
- 스킬에 없는 불필요한 단계를 추가했는가?

**1~10점으로 지침 준수도를 채점**하고 구체적인 문제점을 기록합니다.

#### 5단계: 승자 강점 파악

승자를 더 낫게 만든 요인을 구체적으로 파악합니다:
- 더 나은 행동을 이끈 명확한 지침?
- 더 나은 출력을 생성한 스크립트/도구?
- 엣지 케이스를 안내한 포괄적인 예시?
- 더 나은 오류 처리 지침?

스킬/출력에서 관련 부분을 직접 인용합니다.

#### 6단계: 패자 약점 파악

패자를 뒤처지게 만든 요인을 구체적으로 파악합니다:
- 최적이 아닌 선택으로 이어진 모호한 지침?
- 대안을 찾게 만든 누락된 도구/스크립트?
- 엣지 케이스 처리의 공백?
- 실패를 유발한 미흡한 오류 처리?

#### 7단계: 개선 제안 생성

패자 스킬을 개선하기 위한 실행 가능한 제안을 도출합니다:
- 변경할 구체적인 지침
- 추가하거나 수정할 도구/스크립트
- 포함할 예시
- 다뤄야 할 엣지 케이스

영향도 기준으로 우선순위를 정합니다. 결과를 바꿀 수 있었던 변경사항에 집중합니다.

#### 8단계: 결과 저장

```
write_file(path="<output_path>", content="...")
```

### 분석 원칙

| 원칙 | 설명 |
|------|------|
| 구체적으로 | 스킬/출력에서 직접 인용. "지침이 불명확했다"고만 하지 말 것 |
| 실행 가능하게 | 제안은 막연한 조언이 아닌 구체적인 변경사항이어야 함 |
| 스킬 개선에 집중 | 목표는 패자 스킬 개선. 실행 에이전트 비판이 아님 |
| 영향도 기준 우선순위 | 결과를 바꿀 수 있었을 변경사항이 최우선 |
| 인과관계 고려 | 스킬 약점이 실제로 나쁜 출력을 유발했는가, 우연인가? |
| 일반화 고려 | 이 개선이 다른 eval에서도 도움이 될까? |

### 개선 제안 카테고리

| 카테고리 | 설명 |
|----------|------|
| `instructions` | 스킬의 지침 문구 변경 |
| `tools` | 추가하거나 수정할 스크립트/템플릿/도구 |
| `examples` | 포함할 입출력 예시 |
| `error_handling` | 실패 처리 안내 |
| `structure` | 스킬 내용 재구성 |
| `references` | 추가할 외부 문서/자료 |

### 우선순위 기준

- **high**: 이 비교의 결과를 바꿀 가능성이 높음
- **medium**: 품질을 개선하지만 승패를 바꾸지는 않을 수 있음
- **low**: 있으면 좋지만 미미한 개선

## 역할 2 — 벤치마크 패턴 분석

여러 실행 결과를 검토하여 **집계 지표만으로는 보이지 않는 패턴과 이상값**을 발견합니다. 스킬 개선 제안은 이 단계의 목적이 아닙니다.

### 필요한 정보

사용자가 다음을 제공합니다:
- `sessions/` 또는 별도 저장 경로에 있는 실행 결과 파일들
- 벤치마킹 대상 스킬 경로

---

### 수행 절차

#### 1단계: 실행 결과 읽기

```
read_file(path="<결과 파일 경로>")
```

- 테스트한 구성(스킬 사용/미사용)을 확인합니다.
- 이미 계산된 집계값을 메모합니다.

#### 2단계: 어서션별 패턴 분석

각 어서션에 대해:
- 두 구성 모두에서 **항상 통과** → 변별력 없는 테스트일 수 있음
- 두 구성 모두에서 **항상 실패** → 테스트가 잘못됐거나 모델 능력 초과
- **스킬 사용 시만 통과** → 스킬이 명확한 가치를 추가
- **스킬 사용 시만 실패** → 스킬이 오히려 방해할 수 있음
- **높은 분산** → 불안정하거나 비결정적인 동작

#### 3단계: eval 간 패턴 분석

여러 eval에 걸친 패턴을 확인합니다:
- 특정 eval 유형이 일관되게 더 어렵거나 쉬운가?
- 일부 eval은 높은 분산을 보이고 다른 것은 안정적인가?
- 기대를 뒤집는 놀라운 결과가 있는가?

#### 4단계: 리소스 사용 패턴 분석

실행 시간과 토큰 사용을 확인합니다:
- 스킬이 실행 시간을 크게 증가시키는가?
- 리소스 사용의 분산이 높은가?
- 집계값을 왜곡하는 이상값 실행이 있는가?

#### 5단계: 관찰 노트 작성

각 노트는:
- 구체적인 관찰 사항을 기술합니다.
- 데이터에 근거합니다 (추측이 아님).
- 집계 지표가 보여주지 않는 것을 설명합니다.

**좋은 노트 예시:**
- "어서션 '출력이 PDF 파일임'은 두 구성 모두에서 100% 통과 — 변별력이 없을 수 있음"
- "Eval 3은 높은 분산(50% ± 40%) — 실행 2의 비정상적인 실패가 불안정한 것일 수 있음"
- "스킬 미사용 실행이 테이블 추출 어서션에서 일관되게 실패 (0% 통과율)"
- "스킬이 평균 13초의 실행 시간을 추가하지만 통과율을 50% 개선"
- "토큰 사용량이 스킬 사용 시 80% 증가 — 주로 스크립트 출력 파싱 때문"

#### 6단계: 결과 저장

```
write_file(path="<결과 저장 경로>", content="[\"노트1\", \"노트2\", ...]")
```

---

### 분석 원칙

**해야 할 것:**
- 데이터에서 관찰한 것을 보고합니다
- 어떤 eval, 어서션, 실행을 언급하는지 구체적으로 표시합니다
- 집계 지표가 숨기는 패턴에 주목합니다
- 숫자를 해석하는 데 도움이 되는 맥락을 제공합니다

**하지 말아야 할 것:**
- 스킬 개선을 제안합니다 (그것은 개선 단계의 역할)
- 주관적인 품질 판단을 합니다 ("출력이 좋았다/나빴다")
- 증거 없이 원인을 추측합니다
- 이미 집계값에 있는 정보를 반복합니다
