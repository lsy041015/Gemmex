# Skill Creator — 스킬 생성 및 개선 가이드

새 스킬을 만들거나, 기존 스킬을 수정/개선하거나, 스킬 성능을 측정할 때 사용합니다.
사용자가 스킬을 처음부터 만들거나, 기존 스킬을 편집/최적화하거나, 평가(eval)를 실행해 스킬을 테스트하거나, 분산 분석을 포함한 성능 벤치마크를 측정하거나, 더 나은 트리거 정확도를 위해 스킬 설명을 최적화하려 할 때 활용합니다.

---

## 개요

스킬을 만드는 과정은 크게 다음과 같습니다.

1. 스킬이 무엇을 할지, 대략 어떻게 동작할지 결정
2. 스킬 초안 작성
3. 테스트 프롬프트 몇 개를 만들고, 스킬에 접근 가능한 Gemma로 실행
4. 사용자가 결과를 정성적·정량적으로 평가하도록 지원

실행이 백그라운드에서 진행되는 동안, 아직 정량 평가(eval)가 없다면 초안을 작성합니다. 있다면 검토하거나 필요하면 수정합니다. 사용자에게 eval을 설명한 뒤, `eval-viewer/generate_review.py` 스크립트로 결과를 보여 줍니다.

평가 결과와 정량 지표의 명백한 결함을 바탕으로 스킬을 재작성합니다. 만족할 때까지 반복합니다. 테스트 셋을 확장하고 더 큰 규모로 다시 시도합니다.

이 스킬을 사용할 때는 사용자가 이 과정의 어느 단계에 있는지 파악하고, 적절한 지점에서 진행을 도와야 합니다. 사용자가 "X를 위한 스킬을 만들고 싶다"고 하면 의도를 좁히고, 초안을 작성하고, 테스트 케이스를 만들고, 평가 방식을 정하고, 모든 프롬프트를 실행한 뒤 반복합니다. 반대로 이미 스킬 초안이 있다면 eval/반복 단계로 바로 진입합니다.

사용자가 "평가 없이 그냥 같이 해봐"라고 하면 유연하게 대응해도 됩니다.

---

## 사용자와의 소통

스킬 크리에이터는 코딩에 익숙하지 않은 사람부터 전문가까지 다양한 사용자가 사용합니다. 문맥 단서를 보고 용어 수준을 조절하세요.

- "평가(evaluation)"와 "벤치마크(benchmark)"는 경계선 수준이지만 사용 가능
- "JSON"이나 "어서션(assertion)" 같은 용어는 사용자가 익숙하다는 신호가 보일 때만 사용
- 불확실하면 짧게 설명을 덧붙이는 것이 좋음

## 스킬 만들기

### 1. 의도 파악 (Capture Intent)

사용자의 의도를 먼저 이해합니다. 현재 대화에 이미 캡처하고 싶은 워크플로우가 있다면 (예: "이것을 스킬로 만들어줘"), 대화 기록에서 먼저 정보를 추출합니다 — 사용된 도구, 단계 순서, 사용자가 한 수정, 관찰된 입출력 형식 등.

확인해야 할 사항:
- 이 스킬이 Gemma에게 무엇을 가능하게 해야 하는가?
- 언제 이 스킬이 트리거되어야 하는가? (어떤 문구/맥락에서)
- 예상 출력 형식은 무엇인가?
- 테스트 케이스를 만들어 스킬이 작동하는지 검증해야 하는가?

> 객관적으로 검증 가능한 출력(파일 변환, 데이터 추출, 코드 생성, 고정 워크플로우)을 가진 스킬은 테스트 케이스가 유용합니다. 주관적인 출력(글쓰기 스타일, 예술)은 보통 불필요합니다.

### 2. 인터뷰 및 리서치 (Interview and Research)

엣지 케이스, 입출력 형식, 예시 파일, 성공 기준, 의존성에 대해 적극적으로 질문합니다. 테스트 프롬프트는 이 단계를 마친 후에 작성합니다.

필요하면 `read_file`, `find_files`, `search_in_files` 도구로 관련 문서와 유사 스킬을 조사합니다.

### 3. SKILL.md 작성

인터뷰를 바탕으로 다음 구성요소를 채웁니다:

- `name`: 스킬 식별자
- `description`: 언제 트리거할지, 무엇을 하는지. **이것이 핵심 트리거 메커니즘** — "무엇을 할 때 쓰는지"에 대한 모든 정보를 여기에 담습니다. 본문에는 넣지 않습니다.
  > Gemma는 스킬을 덜 트리거하는 경향이 있으므로, 설명을 조금 "적극적으로" 작성합니다. 예: "사용자가 대시보드, 데이터 시각화, 내부 지표를 언급하거나 데이터를 표시하고 싶어 할 때마다 이 스킬을 사용하세요 — 명시적으로 '대시보드'라고 하지 않더라도."
- `compatibility`: 필요한 도구, 의존성 (선택적, 드물게 필요)
- 나머지 스킬 내용

## 스킬의 구조 (Anatomy of a Skill)

```
skill-name/
├── SKILL.md (필수)
│   ├── YAML 프론트매터 (name, description 필수)
│   └── 마크다운 지침
└── 번들 리소스 (선택)
    ├── scripts/    - 결정론적/반복적 작업을 위한 실행 가능 코드
    ├── references/ - 필요에 따라 컨텍스트로 로드되는 문서
    └── assets/     - 출력에 사용되는 파일 (템플릿, 아이콘, 폰트)
```

### 점진적 공개 (Progressive Disclosure)

스킬은 3단계 로딩 시스템을 사용합니다:

1. **메타데이터** (name + description) — 항상 컨텍스트에 포함 (~100 단어)
2. **SKILL.md 본문** — 스킬이 트리거될 때마다 컨텍스트에 포함 (이상적으로 500줄 이하)
3. **번들 리소스** — 필요할 때만 (무제한, 스크립트는 로드 없이 실행 가능)

핵심 패턴:
- SKILL.md는 500줄 이하로 유지; 한계에 가까워지면 추가 계층 구조와 명확한 포인터를 추가
- 언제 읽어야 하는지 안내와 함께 SKILL.md에서 파일을 명확히 참조
- 큰 참조 파일(300줄 이상)에는 목차를 포함

도메인 조직: 스킬이 여러 도메인/프레임워크를 지원하는 경우 변형별로 구성합니다:
```
cloud-deploy/
├── SKILL.md (워크플로우 + 선택)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

### 최소 놀람의 원칙 (Principle of Lack of Surprise)

스킬에는 맬웨어, 익스플로잇 코드, 시스템 보안을 침해할 수 있는 내용이 포함되어서는 안 됩니다. 잘못된 정보를 제공하거나 무단 접근, 데이터 유출 등 악의적인 활동을 촉진하는 스킬을 만들어서는 안 됩니다.

### 작성 패턴

- 지시사항은 명령형으로 작성
- 출력 형식 정의 예시:

```markdown
## 보고서 구조
항상 이 정확한 템플릿을 사용합니다:
# [제목]
## 요약
## 주요 발견사항
## 권고사항
```

- 예시 패턴:

```markdown
## 커밋 메시지 형식
**예시 1:**
입력: JWT 토큰으로 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현
```

### 작성 스타일

- 무거운 MUST 대신 "왜 중요한지"를 설명
- 이론적 추론을 사용하여 스킬을 특정 예시에 과적합하지 않고 일반적으로 만들기
- 초안을 작성한 후 새로운 시각으로 검토하고 개선

## 테스트 케이스

스킬 초안 작성 후 2-3개의 현실적인 테스트 프롬프트를 만듭니다. 사용자에게 공유: "이런 테스트 케이스를 시도해보고 싶습니다. 괜찮으신가요, 아니면 더 추가하고 싶으신가요?"

테스트 케이스를 `evals/evals.json`에 저장합니다. 어서션은 아직 작성하지 않습니다 — 다음 단계에서 작성합니다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자의 작업 프롬프트",
      "expected_output": "예상 결과 설명",
      "files": []
    }
  ]
}
```

## 테스트 케이스 실행 및 평가

이 섹션은 하나의 연속적인 순서입니다 — 중간에 멈추지 마세요.

결과는 스킬 디렉토리의 형제로 `<skill-name>-workspace/`에 저장합니다. 반복(iteration)별로 구성합니다 (`iteration-1/`, `iteration-2/` 등).

### 1단계: 어서션 초안 작성

실행 전 각 테스트 케이스에 대한 `eval_metadata.json`을 작성합니다. 좋은 어서션은 객관적으로 검증 가능하며 설명적인 이름을 가집니다. 주관적인 스킬(글쓰기 스타일, 디자인 품질)은 정성적 평가가 더 적합합니다.

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "사용자의 작업 프롬프트",
  "assertions": [
    {"text": "출력 파일이 생성됐는가", "passed": false, "evidence": ""}
  ]
}
```

### 2단계: 테스트 케이스 순차 실행

각 테스트 케이스를 한 번에 하나씩 직접 수행합니다. 스킬 지침(`SKILL.md` 또는 참조 파일)을 `read_file`로 읽고 지침에 따라 작업을 완수합니다.

출력은 `<workspace>/iteration-N/eval-<ID>/with_skill/outputs/`에 저장하고, 실행 시작·종료 시각을 기록합니다:

```python
from datetime import datetime
started_at  = datetime.now().isoformat()
# ... 작업 수행 ...
finished_at = datetime.now().isoformat()
```

이전 반복이 있다면 (`iteration-1/` 등) 비교 기준으로 활용합니다. 별도 베이스라인 실행은 없습니다.

### 3단계: 채점

각 실행을 완료하는 즉시 인라인으로 채점하고 `grading.json`을 저장합니다. `expectations` 배열은 반드시 `text`, `passed`, `evidence` 필드를 포함해야 합니다.

```json
{
  "summary": {"passed": 2, "failed": 1, "total": 3, "pass_rate": 0.67},
  "expectations": [
    {"text": "출력 파일이 생성됐는가", "passed": true,  "evidence": "output.csv 확인"},
    {"text": "헤더 행이 포함됐는가",   "passed": true,  "evidence": "첫 줄: name,value"},
    {"text": "빈 행이 없는가",         "passed": false, "evidence": "3번째 행 빈 줄 발견"}
  ],
  "timing": {
    "started_at": "2025-01-15T10:00:00",
    "finished_at": "2025-01-15T10:00:45",
    "duration_seconds": 45.0
  }
}
```

### 4단계: 집계 및 뷰어 생성

모든 실행이 완료되면:

1. **벤치마크로 집계** — 스킬 크리에이터 디렉토리에서 실행:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   `benchmark.json`과 `benchmark.md` 생성.

2. **분석가 패스** — 패턴 파악: 항상 통과하는 어서션(비변별적), 높은 분산의 eval(불안정 가능성).

3. **정적 뷰어 생성**:
   ```bash
   python skill-creator/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     --static /tmp/review_<skill>_iter<N>.html
   ```
   반복 2 이상에서는 `--previous-workspace <workspace>/iteration-<N-1>` 추가.

> **중요**: 뷰어를 생성한 후 직접 결과를 평가하세요. 수정 전에 반드시 사용자에게 먼저 보여줍니다.

### 5단계: 피드백 수집

사용자에게 결과를 TUI 대화에서 직접 제시하고 인라인으로 피드백을 받습니다. 뷰어 HTML을 열어 확인할 수도 있습니다.

`feedback.json`이 있으면 읽습니다:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "차트에 축 레이블이 없음"},
    {"run_id": "eval-1-with_skill", "feedback": ""},
    {"run_id": "eval-2-with_skill", "feedback": "완벽합니다"}
  ],
  "status": "complete"
}
```

빈 피드백 = 사용자가 만족함. 구체적인 불만이 있는 테스트 케이스에 집중합니다.

## 스킬 개선

### 개선에 대한 사고 방식

- **피드백에서 일반화**: 스킬은 수많은 다양한 프롬프트에서 사용될 것입니다. 지금 반복하는 몇 가지 예시에만 작동하는 스킬은 무용합니다. 고집스러운 문제가 있다면 다른 비유나 패턴을 시도해보세요.
- **프롬프트를 간결하게**: 효과가 없는 요소를 제거합니다. 트랜스크립트를 읽고 — 최종 출력만이 아니라 — 모델이 비생산적인 일에 시간을 낭비하고 있다면 그 부분을 제거해보세요.
- **이유를 설명**: 요청하는 모든 것에 대해 이유를 설명합니다. ALWAYS나 NEVER를 대문자로 쓰고 싶어진다면 경고 신호 — 대신 그것이 왜 중요한지 설명하세요.
- **테스트 케이스 간 반복 작업 찾기**: 여러 eval에서 Gemma가 같은 도우미 코드를 반복 작성하고 있다면, 그 코드를 스킬 번들에 포함해야 한다는 신호입니다.

### 반복 루프

개선 후:
1. 스킬에 개선사항 적용 (`write_file`로 저장)
2. 모든 테스트 케이스를 새 `iteration-<N+1>/` 디렉토리에서 재실행
3. `--previous-workspace`로 이전 반복과 비교하는 뷰어 생성
4. 사용자에게 결과 제시 및 피드백 수집
5. 새 피드백을 읽고, 다시 개선하고, 반복

종료 조건:
- 사용자가 만족한다고 말할 때
- 피드백이 모두 비어 있을 때 (모든 것이 좋아 보임)
- 의미 있는 진전이 없을 때

### 고급: 블라인드 비교 (Advanced: Blind Comparison)

두 반복의 출력을 어느 것이 어느 버전인지 밝히지 않고 Gemma 스스로 평가하게 합니다. 두 출력 파일을 `read_file`로 읽은 뒤 `agent/comparator.md` 지침에 따라 채점합니다. 패턴 분석은 `agent/analyzer.md`를 참고하세요.

## 설명 최적화 (Description Optimization)

스킬을 만들거나 개선한 후, 더 나은 트리거 정확도를 위해 설명을 최적화하는 것을 제안합니다.

### 1단계: 트리거 eval 쿼리 생성

20개의 eval 쿼리를 만듭니다 — should-trigger와 should-not-trigger 혼합:

```json
[
  {"query": "사용자 프롬프트", "should_trigger": true},
  {"query": "다른 프롬프트", "should_trigger": false}
]
```

쿼리는 현실적이어야 합니다. 예시:

- **나쁜**: "이 데이터를 형식화해줘", "PDF에서 텍스트 추출"
- **좋은**: "상사가 'Q4 sales final FINAL v2.xlsx' 파일을 보내서 수익 마진 퍼센트를 보여주는 열을 추가해야 합니다. 수익은 C열, 비용은 D열인 것 같아요"

should-not-trigger 쿼리의 핵심: 명백히 관련 없는 쿼리는 피하고, 스킬과 키워드나 개념을 공유하지만 실제로는 다른 것이 필요한 근접 미스(near-miss)를 만드세요.

### 2단계: 사용자와 검토

HTML 템플릿으로 eval 셋을 사용자에게 제시:
1. `assets/eval_review.html` 템플릿 읽기
2. 플레이스홀더 교체:
   - `__EVAL_DATA_PLACEHOLDER__` → JSON 배열
   - `__SKILL_NAME_PLACEHOLDER__` → 스킬 이름
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 현재 설명
3. `/tmp/eval_review_<skill-name>.html`에 저장하고 열기
4. 사용자가 편집, 토글, 추가/제거 후 "Export Eval Set" 클릭
5. `~/Downloads/eval_set.json` 확인

### 3단계: 수동 최적화

Gemma TUI는 자동화 최적화 스크립트를 지원하지 않습니다. 다음 절차로 수동 최적화합니다.

1. eval 쿼리를 Gemma에게 직접 입력하고 해당 스킬/섹션이 의도대로 활성화되는지 확인
2. 트리거 실패 쿼리를 모아 공통 패턴 파악
3. 파악한 패턴을 바탕으로 스킬 설명(description) 수정
4. 수정된 설명으로 eval 쿼리를 다시 실행하고 개선 여부 확인
5. 만족할 때까지 반복

**스킬 트리거 작동 방식:** Gemma는 시스템 프롬프트(`GEMMA_MANUAL.md`)에 정의된 지침과 현재 대화 맥락을 바탕으로 어떤 섹션/스킬을 적용할지 결정합니다. 설명이 구체적이고 트리거 조건이 명확할수록 올바른 섹션이 활성화될 가능성이 높아집니다. eval 쿼리는 실제 사용자가 입력할 법한 충분히 구체적인 내용이어야 합니다.

### 4단계: 결과 적용

개선된 설명을 `GEMMA_MANUAL.md` 또는 해당 스킬 섹션에 반영합니다. 사용자에게 변경 전/후를 비교해 보여줍니다.

## 패키징 (Packaging)

`present_files` 도구에 접근 가능한 경우:

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

패키징 후 사용자에게 `.skill` 파일 경로를 알려줍니다.

## 핵심 루프 요약

1. 스킬이 무엇에 관한 것인지 파악
2. 스킬 초안 작성 또는 편집 (`GEMMA_MANUAL.md` 또는 별도 스킬 파일)
3. Gemma에게 테스트 프롬프트를 직접 입력하여 실행
4. 사용자와 함께 출력을 TUI에서 정성적으로 평가하고 피드백 수집
5. 피드백을 반영해 스킬 수정, 만족할 때까지 반복
6. 완성된 스킬을 `write_file`로 저장

## 참조 파일

- `GEMMA_MANUAL.md` — Gemma 시스템 프롬프트 (역할, 능력, 행동 규칙 정의)
- `Skill.md` — 스킬 생성 및 개선 가이드 (현재 파일)
- `sessions/` — 저장된 대화 세션 (JSON 형식, `/save` 명령으로 생성)
