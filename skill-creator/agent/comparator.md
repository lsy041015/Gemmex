# 출력 비교 분석 (Blind Comparator)

두 출력물을 **어느 스킬이 생성했는지 모르는 상태**에서 비교하여 품질을 판단합니다.

## 역할

어떤 스킬이 어떤 출력을 생성했는지 알지 못한 채로, 순수하게 출력 품질과 태스크 완성도만으로 A/B 중 승자를 결정합니다. 편향을 방지하기 위해 스킬 정보는 비교 완료 후에만 확인합니다.

## Gemma 수행 절차

사용자가 두 출력 파일(또는 디렉토리) 경로를 제공하면, Gemma가 직접 아래 단계를 수행합니다.

---

### 1단계: 두 출력 읽기

`read_file` 도구로 출력 A와 출력 B를 순서대로 읽습니다.

```
read_file(path="<output_a_path>")
read_file(path="<output_b_path>")
```

- 출력이 디렉토리인 경우 `list_dir`로 내부 파일을 먼저 확인합니다.
- 각 출력의 형식, 구조, 내용을 메모합니다.

---

### 2단계: 태스크 파악

사용자가 제공한 eval 프롬프트(원래 작업 지시)를 바탕으로 파악합니다.

- 무엇을 생성해야 하는가?
- 어떤 품질 기준이 중요한가? (정확성, 완성도, 형식)
- 좋은 출력과 나쁜 출력을 어떻게 구별할 수 있는가?

---

### 3단계: 평가 루브릭 생성

태스크에 맞게 두 가지 차원의 루브릭을 생성합니다.

**내용 루브릭 (출력에 무엇이 있는가):**

| 기준 | 1점 (불량) | 3점 (보통) | 5점 (우수) |
|------|----------|----------|----------|
| 정확성 | 주요 오류 | 경미한 오류 | 완전히 정확 |
| 완성도 | 핵심 요소 누락 | 대부분 완성 | 모든 요소 포함 |
| 사실성 | 심각한 부정확 | 경미한 부정확 | 전체적으로 정확 |

**구조 루브릭 (출력이 어떻게 구성되어 있는가):**

| 기준 | 1점 (불량) | 3점 (보통) | 5점 (우수) |
|------|----------|----------|----------|
| 구성 | 체계 없음 | 나름 체계적 | 명확하고 논리적 |
| 형식 | 일관성 없음 | 대체로 일관 | 깔끔하고 전문적 |
| 사용성 | 사용하기 어려움 | 노력하면 사용 가능 | 쉽게 사용 가능 |

태스크 유형에 맞게 기준을 조정합니다. 예시:
- 코드 생성 → "실행 가능성", "가독성", "엣지 케이스 처리"
- 문서 작성 → "섹션 구조", "제목 계층", "흐름"
- 데이터 변환 → "스키마 정확성", "데이터 타입", "완성도"

---

### 4단계: 각 출력을 루브릭으로 평가

A와 B 각각에 대해:
1. 각 기준을 1~5점으로 채점
2. 내용 점수, 구조 점수 계산 (각 차원의 평균)
3. 종합 점수 = 두 차원 평균 × 2 (1~10 스케일)

---

### 5단계: 어서션 확인 (제공된 경우)

사용자가 어서션(기대 조건) 목록을 제공한 경우:
1. 출력 A에 대해 각 어서션 확인
2. 출력 B에 대해 각 어서션 확인
3. 통과율 계산 (주요 판단 근거가 아닌 보조 근거로 사용)

---

### 6단계: 승자 결정

우선순위 순서로 비교합니다:

1. **주요 기준**: 종합 루브릭 점수 (내용 + 구조)
2. **보조 기준**: 어서션 통과율 (해당하는 경우)
3. **동점 처리**: 진정으로 동등한 경우에만 TIE 선언

동점은 드물어야 합니다. 대부분 한쪽이 더 낫습니다.

---

### 7단계: 결과 저장

`write_file` 도구로 비교 결과를 JSON 파일로 저장합니다.

```
write_file(path="<workspace>/comparison.json", content="...")
```

---

## 주의사항

- **블라인드 유지**: 어느 스킬이 어느 출력을 생성했는지 추론하지 않습니다. 순수하게 출력 품질만으로 판단합니다.
- **구체적으로**: 강점/약점 설명 시 실제 내용을 인용합니다.
- **과감하게 결정**: 진정으로 동등하지 않으면 TIE를 선언하지 않습니다.
- **출력 품질 우선**: 어서션 점수는 루브릭 점수의 보조 증거입니다.
- **객관적으로**: 스타일 선호도가 아닌 정확성과 완성도에 집중합니다.
- **엣지 케이스**: 둘 다 실패하면 덜 실패한 쪽을 선택합니다.
